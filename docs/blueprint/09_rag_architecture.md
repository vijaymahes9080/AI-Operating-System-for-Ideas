# RAG Architecture & Ingestion Pipeline

IdeasOS uses a **Hybrid Retrieval-Augmented Generation (RAG)** pipeline to query ingested research papers, repositories, databases, web captures, and user transcripts. The RAG architecture guarantees that every response generated by IdeasOS is backed by real sources.

---

## 1. Document Ingestion Pipeline

The ingestion pipeline handles raw file uploads (PDF, DOCX, MD, PNG/JPG, MP3/WAV, GitHub URLs) and processes them sequentially:

```
[Uploaded File] 
       |
       v
[MIME Classifier] 
       |
       +---> (PDF/Text)   ---> [Apache Tika / Unstructured] ---> [Semantic Parser]
       |
       +---> (Image/Draw) ---> [Tesseract OCR / OpenCLIP]   ---> [Layout Parser]
       |
       +---> (Audio/Voice)---> [Whisper Local API]          ---> [Transcript Parser]
       |
       v
[Raw Markdown Document]
       |
       v
[Dynamic Chunking Node]
       |
       v
[Embedder & Indexer (SQLite-vec / Qdrant)]
```

---

## 2. Chunking Strategy

Rather than using basic character splits, IdeasOS employs a dual-mode chunking mechanism:

### A. Hierarchical Parent-Child Chunking
- **Parent Chunk**: Standard semantic sections (e.g. Markdown headers, PDF chapters) of 1,000–2,000 tokens. Retained for context injection during final synthesis.
- **Child Chunk**: Sentences or paragraphs of 150–250 tokens nested inside the parent. Used for search vector lookups. When a child chunk matches, the parent chunk is passed to the LLM to provide surrounding context.

### B. Semantic Sentence Windowing
- Chunks are split by logical sentence boundaries. When a specific sentence matches a query, a window of $N$ sentences before and after is included to preserve flow.

---

## 3. Hybrid Search, Reranking, and Citations

To ensure high-precision search matches:

```
                           [User Query]
                             /      \
                            /        \
                    (Dense Search)  (Sparse Search)
                     SQLite-vec       BM25 Engine
                            \        /
                             \      /
                           [Reciprocal Rank Fusion]
                                    |
                           [Top 50 Candidates]
                                    |
                           [BGE-Reranker-v2]
                                    |
                           [Top 5 Citations]
                                    v
                           [Context Builder]
```

1. **Dense Vector Search**: Computes embeddings using `bge-small-en-v1.5` or `Gemma-2b-it` (via Ollama/LiteLLM), matching semantic concepts.
2. **Sparse Text Search**: Run in parallel using a **BM25** index built over the text strings. Matches exact terms (e.g. specific function names, API paths, or patent numbers).
3. **RRF (Reciprocal Rank Fusion)**: Merges scores from dense and sparse queries to return the top 50 candidate chunks.
4. **Cross-Encoder Reranking**: Re-evaluates top candidates using the local **BGE-Reranker-v2** model to output the final top 5 context snippets.
5. **Strict Citation Engine**:
   - Every chunk database record contains a `source_hash` and `document_id`.
   - The prompt forces the LLM to format citations using Markdown links referencing the metadata record (e.g. `[DeepSeek-V3 Research, p. 12](file:///docs/inbox/deepseek_v3.pdf#page=12)`).
   - A post-processing hook validates that every citation link matches a real document inside the local SQLite database.
